# 代码审查（提审 + 受审两面）

完成一块功能、合并前，或拿到别人/AI 的审查意见时，用本技能把关。目标是**抓真问题**，不是表演式点赞或盲改。

## 当审查者（review 别人/自己的 diff）
按优先级看：
1. **正确性**：边界条件、空值 / 错误处理、并发与竞态、资源泄漏、可逆/破坏性操作有没有护栏。
2. **安全**：注入（SQL / 命令 / 路径穿越）、鉴权、机密硬编码、不可信输入当指令执行。
3. **复用与简化**：有没有重复造轮子、能不能用现有函数、有没有过度设计。
4. **一致性**：是否贴合周边代码的命名 / 注释密度 / 习惯。
- 每条意见给 `file:line` + 「问题是什么 + 为什么 + 建议怎么改」。区分**必改**与**可选**。
- 没把握的标注「存疑」，别当确凿结论。

## 当被审者（收到意见时）
- **先验证，再动手**：意见也可能是错的。看不懂或技术上存疑的，先核实再改，别盲从也别敷衍。
- 同意的就改并说清；不同意的讲清理由；拿不准的回问。
- 别为了显得配合而假装认同 —— 技术严谨高于客气。

## 约定
- 审查针对代码不针对人。
- 改完按「完成前验证」跑一遍再说处理好了。
