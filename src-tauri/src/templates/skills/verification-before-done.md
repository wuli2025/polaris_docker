# 完成前验证（先有证据，再说做好了）

在宣称「完成 / 修好 / 通过」、或提交、建 PR 之前，**先真的跑一遍验证命令，看到通过的输出**，再下结论。证据先于断言。

## 必做
1. **跑该跑的**：编译 / 构建 / 类型检查 / 测试 / lint —— 项目里有什么就跑什么，看退出码和输出。
2. **跑核心路径**：改了功能就实际触发一次（命令行跑、或真机点一遍），别只靠"代码看着对"。
3. **如实报告**：
   - 测试失败 → 就说失败，贴关键输出。
   - 跳过了某步 → 明说跳过了。
   - 真验过且通过 → 平实地说清楚验了什么、结果如何，不浮夸。

## 红线
- 没跑验证就说"应该没问题 / 一般来说可以"。
- 测试报红却说"功能已完成"。
- 用"理论上"代替"实测"。

## 对 Polaris 的现实
本项目大半工作没法用单测覆盖。验证的现实组合 = `cargo check`（两 flavor）+ `vue-tsc` + **真机点一遍**。三件里能做的都要做；真机没点测就如实标注「真机未点测」，别含糊带过。
