# 写实现计划（动手前先成文）

面对**多步骤 / 跨文件 / 有不确定性**的任务，先写一份计划再动代码。一句话能改完的别写计划。

## 何时触发
- 任务涉及 ≥3 个步骤或多个文件改动
- 有架构取舍、迁移、重构、或牵一发动全身的改动
- 需求里有歧义、或有多种合理实现路线

## 计划要包含什么
1. **目标与验收**：做完什么算成功？怎么验证（命令 / 真机操作 / 输出）。
2. **现状勘查**：先读相关代码，列出受影响的文件、函数、调用链 —— 用 `file:line` 标注，别凭记忆。
3. **分步骤**：每步「改哪个文件、做什么、为什么」，按依赖排序；标出可并行的步骤。
4. **风险与回滚**：最可能踩的坑、不可逆动作、出错怎么退。
5. **明确不做**：划清边界，防止范围蔓延。

## 约定
- 计划是**可执行清单**不是作文：每条都能直接照着改。
- 不确定的点先标 `[待定]` 并给出倾向方案，别假装已定。
- 计划获认可后再动手；执行中发现计划错了，先更新计划再改代码。

## 不适用
单文件小改、改文案、改样式、重命名 —— 直接做，不必先写计划。
