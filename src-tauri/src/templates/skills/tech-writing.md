# 技术文档（README / API 文档 / Changelog）

把代码 / 项目写成别人读得懂、用得起来的文档。读者视角优先：他想快速跑起来、查清楚怎么用。

## README
结构（按需取舍）：
1. **一句话讲清这是什么、解决什么问题**。
2. **快速开始**：装依赖 → 配置 → 跑起来，给可直接复制的命令。
3. **用法**：最常见的 2-3 个场景 + 例子。
4. **配置**：环境变量 / 选项表（名称 / 默认值 / 说明）。
5. 可选：架构概览、目录结构、常见问题、贡献指南、许可证。
- 命令给完整可跑的，标清平台差异（Win / mac / Linux）。

## API 文档
- 每个端点 / 函数：用途、入参（类型 / 必填 / 默认）、返回、错误情形、一个真实例子。
- 接口契约优先于实现细节；变更要同步更新文档。
- HTTP API 可用 OpenAPI/Swagger 规范化。

## Changelog
- 遵循 Keep a Changelog：按版本倒序，分 `Added / Changed / Fixed / Removed / Deprecated / Security`。
- 写「对用户的影响」，不是 commit 流水账；破坏性变更醒目标注。

## 约定
- 写真话：没实现的别写成已支持；有坑的标清楚。
- 例子能跑通；命令 / 路径 / 版本号要准。
- 简洁有结构，砍废话；能用表格 / 代码块就别堆长句。
