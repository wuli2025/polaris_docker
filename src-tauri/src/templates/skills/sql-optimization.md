# SQL 查询优化

让慢查询变快。先量后改，别凭感觉加索引。

## 流程
1. **先测**：用 `EXPLAIN` / `EXPLAIN ANALYZE`（PostgreSQL）/ `EXPLAIN` + `SHOW PROFILE`（MySQL）看真实执行计划，找全表扫描、临时表、文件排序、嵌套循环吃满行数的地方。
2. **定位瓶颈**：是缺索引？索引没命中？回表太多？JOIN 顺序差？数据量本身就大？
3. **针对性改**，每改一处再 `EXPLAIN` 对比。

## 常见手法
- **加对索引**：在 `WHERE` / `JOIN` / `ORDER BY` 的列上建；多列查询用**复合索引**且注意列顺序（最左前缀）。
- **覆盖索引**：把 `SELECT` 的列也纳入索引，免回表。
- **别 `SELECT *`**：只取需要的列。
- **让索引可用**：避免在索引列上套函数 / 运算 / 隐式类型转换（`WHERE DATE(col)=...` 会让索引失效）。
- **分页**：大偏移 `LIMIT 100000,20` 很慢，改游标分页（`WHERE id > ? LIMIT 20`）。
- **N+1**：循环里逐条查 → 改成一次 `IN` / `JOIN` 批量取。
- **统计信息**：`ANALYZE` 让优化器拿到新的数据分布。

## 约定
- 索引不是越多越好：写入会变慢、占空间。只为真实热点查询加。
- 改完用真实数据量验证提升（别只在小表上看着快）。
- 复杂聚合 / 报表考虑物化视图或预聚合表。
